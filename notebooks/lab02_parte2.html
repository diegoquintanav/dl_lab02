Se reescribió la función debido a la sugerencia contenida en la nueva versión de la API de keras:

/home/diego/anaconda3/envs/tensor/lib/python3.6/site-packages/ipykernel/__main__.py:1: UserWarning: The `input_dim` and `input_length` arguments in recurrent layers are deprecated. Use `input_shape` instead. if __name__ == '__main__': /home/diego/anaconda3/envs/tensor/lib/python3.6/site-packages/ipykernel/__main__.py:1: UserWarning: Update your `SimpleRNN` call to the Keras 2 API: `SimpleRNN(5, return_sequences=False, input_shape=(4, 1))` if __name__ == '__main__':

En este caso, se tiene que

  • El primer término según la documentación corresponde a units, lo que sería la dimensionalidad de $h_t=5$.
  • input_dim es el equivalente a la dimensión de los datos $x_t$ de entrada. En este caso es la demanda mensual, la cual sólo requiere de una dimensión.
  • input_length es el equivalente al largo de la secuencia de entrada. Para este caso, se consideran cuatro meses por lo que su valor es 4
  • El término return_sequences=False indica que el valor la salida $y_t$ se calcula sólo en la última unidad

En términos generales la cantidad de parámetros viene dada por la relación

$$ 4d(n+d) $$

Donde $n$ es la dimensión de $x$ y $d$ es la cantidad de unidades.

Finalmente, dado que se desea predecir una variable de la misma dimensionalidad que la entrada, basta con agregar una capa densa de una neurona, sin función de activación

Actividad 10¶

Se crea el siguiente script en la carpeta ./scripts/AirlinePrediction.py. Dado que el resultado es extenso, se implementa un logger y donde se asignarán letras a cada ensayo, tal que

  • A es la red original
  • B es la red con history=10, es decir input_shape(10,1)
  • C es la red con Dense(100) antes de Dense(1)
  • D es la red con Dropout(0.5) antes de Dense(1)

Se omite el output debido a que el código está diseñado para correr usando CLI, y los resultados pueden verse en el archivo trace.log.

#%load ../scripts/AirlinePrediction.py
from keras.models import Sequential
from keras.layers import Dense, Dropout, TimeDistributed, Flatten, SimpleRNN,LSTM
import airlineUtils
import logging
from logger import set_logger

set_logger()
logger = logging.getLogger(__name__)


logger.debug('Test case: SimpleRNN')

for case in  ['a','b','c','d']: 

    logger.debug('Running case %s',case)
    #set sequence length
    if case == 'b': 
        history=10
    else:
        history=4

    #read training and test sets
    trainX, trainY, testX, testY, scaler, dataset=airlineUtils.readAirlineData(history)

    # create and fit the LSTM network
    modelRNN = Sequential()
    modelRNN.add(SimpleRNN(5,input_shape=(history,1), return_sequences=False))

    if case=='c':
        modelRNN.add(Dense(100)) # ACTIVIDAD 10

    if case=='d':
        modelRNN.add(Dropout(0.5))

    # 
    modelRNN.add(Dense(1))

    #Train model
    modelRNN.compile(loss='mean_squared_error', optimizer='adam')
    modelRNN.fit(trainX, trainY, epochs=150, batch_size=5, verbose=2)

    # Make predictions
    trainPredict = modelRNN.predict(trainX)
    testPredict = modelRNN.predict(testX)

    #Display results
    airlineUtils.displayResult(dataset, trainPredict, trainY, testPredict, testY, scaler, history, filename='simplernn_'+case)

Conclusiones¶

Se observa que el caso B y C tienen un rendimiento mejor, mientras que el caso D tiene el peor rendimiento de todos. Ambos implican modificaciones en la dimensionalidad de parámetros internos.

En el caso de B, history=10 implica según el código utilizado, que la cantidad de meses utilizados varía de 4 a 10, o dicho de otra forma la dimensionalidad de $h_t$ aumenta. a Para el caso de C, se transforma el espacio de salida de $5\times1$ previo a un espacio de $100\times1$, lo cual añade más estados internos a la red. Esto permite un mapeo más complejo de las características de los datos ingresados.

En el caso de D, un dropout por sí solo añade ruido que no aporta a disminuir el error, dada la cantidad de épocas consideradas.

Actividad 12¶

Considerando

modelRNN.fit(trainX, trainY, nb epoch=150, batch size=5, verbose=2)

De la documentación

batch_size: Integer or None. Number of samples per gradient update. If unspecified, it will default to 32.

epochs: Integer. Number of epochs to train the model. An epoch is an iteration over the entire x and y data provided. Note that in conjunction with initial_epoch, epochs is to be understood as "final epoch". The model is not trained for a number of iterations given by epochs, but merely until the epoch of index epochs is reached.

batch size se refiere a la cantidad de datos del dataset que son usados para actualizar el gradiente durante una época. Un tamaño menor de muestra permite un uso menor de memoria, aunque la trayectoria del gradiente presenta más ruido. Un caso es cuando batch_size=1 el cual se conoce como descenso estocástico, el cual presenta el máximo de ruido.

epochs por otro lado, se refieren a las veces que el algoritmo entrena la red usando todo el conjunto de datos. Una época ocurre cuando la red ha visto todo el conjunto de datos. Considerando que al comienzo de cada época los datos son reordenados aleatoriamente, la red ve en cada vez un conjunto distinto y aprende un problema distinto. El tradeoff ocurre considerando que el entrenamiento es supervisado, i.e. a mayor cantidad de épocas la red puede perder la capacidad de generalización y viceversa.

Epochs y Batch Size son hiperparámetros de la red. Una discusión al respecto puede leerse aquí y aquí

Cambios en los hiperparámetros de la red LSTM¶

Si se entrena la red LSTM para el caso C, cambiando los hiperparámetros de la siguiente forma

modelRNN.fit(trainX, trainY, epochs=10, batch_size=100, verbose=2)

se tiene una red con pocas capacidades de generalizar, que obtiene un error relativamente bajo durante el entrenamiento pero muy alto durante la etapa de test, durante la cual la red trabaja sobre elementos nuevos.

modelRNN.fit(trainX, trainY, epochs=100, batch_size=20, verbose=2)

La red tiene mejores resultados durante la etapa de test.

DEBUG:airlineUtils:Train Score: 20.38 RMSE DEBUG:airlineUtils:Test Score: 62.14 RMSE

En este caso las mejoras son pocas, pero el tiempo de entrenamiento se triplica. Se puede observar que la red aprende los ejemplos de entrenamiento logrando una copia casi idéntica. Esto puede ser útil para algunos problemas, y en este caso termina aportando al error.
